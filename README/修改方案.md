# 多模态检索系统 局域网部署 改造方案

> 适用版本：当前仓库（`frontend/` + `backend/` + `algorithm/`）
> 目标场景：**算法服务 + 后端服务**部署在局域网算力服务器上，**演示机仅运行前端**（浏览器或 `npm run dev`）。
> 参考文档：[`README/lan_deployment_plan.md`](README/lan_deployment_plan.md)、[`README/frontend_README.md`](README/frontend_README.md)、[`README/backend_README.md`](README/backend_README.md)、[`README/algorithm_README.md`](README/algorithm_README.md)

---

## 1. 现状梳理

### 1.1 三端默认行为

| 组件 | 监听地址 | 关键默认值 | 是否可直接局域网访问 |
| --- | --- | --- | --- |
| 前端 ([`frontend/vite.config.ts`](frontend/vite.config.ts:1)) | `0.0.0.0:5173`（dev 默认仅 localhost） | `VITE_BACKEND_PROXY_TARGET=http://127.0.0.1:8000` | 需开 `host` |
| 前端 axios ([`frontend/src/api/client.ts`](frontend/src/api/client.ts:3)) | — | `baseURL = VITE_API_BASE_URL \|\| '/'` | 走 Vite 代理或直连 |
| 后端 ([`backend/app/main.py`](backend/app/main.py:25)) | `uvicorn` 默认 `127.0.0.1:8000` | 未启用 CORS | ❌ 局域网访问被拒 |
| 后端 ([`backend/app/core/config.py`](backend/app/core/config.py:14)) | — | `public_base_url=http://127.0.0.1:8000`、`algorithm_gateway_url=http://127.0.0.1:18080` | ❌ 演示机拿到的 `preview_url` 无效 |
| 算法 gateway ([`algorithm/launcher/run_all.py`](algorithm/launcher/run_all.py:17)) | `0.0.0.0:18080` | gateway 内部转发 worker 走 `127.0.0.1:18081/2/3` | ✅ 已监听全网卡 |

### 1.2 当前痛点

1. **后端默认只监听 127.0.0.1**，且**没有 CORS**，演示机浏览器无法直接调 `/api/v1/*`。
2. **`MMR_PUBLIC_BASE_URL` 写死 `127.0.0.1:8000`**，`/api/v1/search`、`/uploads/query-image` 返回的 `preview_url` 在演示机浏览器里指向**演示机自身的 8000 端口**，永远 404。参考 [`backend/app/services/object_service.py`](backend/app/services/object_service.py:17)、[`backend/app/services/search_service.py`](backend/app/services/search_service.py:124)。
3. **算法 worker 读取的是“后端机本地路径”**：`prepare_service` 把媒体落到 `data/assets/...` 后把**绝对路径**作为 `key` 传给算法 `/encode`（见 [`backend/app/services/algorithm_service.py`](backend/app/services/algorithm_service.py:18) 与 [`README/algorithm_README.md`](README/algorithm_README.md:1) 第 10.8 节）。当**后端 ≠ 算法机**时，算法机无法读取到这些文件。
4. **前端 axios 默认 baseURL = `'/'`**，dev 模式依赖 Vite 代理；生产打包后若直接打开静态文件页面、或 dev server 跑在演示机却要访问局域网后端，必须显式指定 `VITE_API_BASE_URL` 或对应代理目标。
5. `run.sh` 是顺序串行命令、且夹带 Windows 路径备忘，**不可作为部署脚本**直接使用。

## 2. 总体架构（目标态）

```
┌────────────────┐        HTTP        ┌──────────────────────────────────────┐
│  演示机        │ ─────────────────▶ │ 局域网服务器 (LAN_HOST)              │
│  浏览器/dev    │                    │  ┌──────────────────────────────┐    │
│  Vite :5173    │                    │  │ Backend  uvicorn :8000       │    │
│                │ ◀───── JSON ────── │  │  (data/assets, data/sqlite)  │    │
└────────────────┘                    │  └──────────┬───────────────────┘    │
        │                             │             │ HTTP                   │
        │ 直接拉媒体                   │  ┌──────────▼───────────────────┐    │
        │ /api/v1/media/preview        │  │ Algorithm Gateway :18080     │    │
        ▼                             │  │  ├─ i2i :18081               │    │
   后端 :8000                          │  │  ├─ t2i :18082               │    │
                                      │  │  └─ t2v :18083               │    │
                                      │  └──────────────────────────────┘    │
                                      └──────────────────────────────────────┘
```

要点：

- **后端与算法服务部署在同一台机器**（或至少同一个本地文件系统），保证算法 worker 读 `data/assets/**` 即可。
- 前端只跟**后端 8000**对话，预览媒体也走后端 `/api/v1/media/preview` 反代。
- 前端 → 算法 **不直连**（与文档 4.1 规定一致）。

## 3. 改造清单（按文件）

### 3.1 后端：开启 CORS、统一可配置域名

#### ① [`backend/app/main.py`](backend/app/main.py:1) 增加 CORS 中间件

```python
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=settings.cors_allow_origins,   # 见下文 config 改造
    allow_credentials=False,
    allow_methods=["*"],
    allow_headers=["*"],
)
```

#### ② [`backend/app/core/config.py`](backend/app/core/config.py:1) 增加 CORS 与监听项

```python
host: str = "0.0.0.0"
port: int = 8000
cors_allow_origins: list[str] = Field(
    default_factory=lambda: ["*"]   # 演示阶段允许全部；正式可改为 ["http://<demo-host>:5173"]
)
```

> 通过环境变量 `MMR_CORS_ALLOW_ORIGINS='["http://10.0.0.5:5173"]'` 注入，pydantic-settings 会自动解析 JSON。

#### ③ [`backend/app/core/config.py`](backend/app/core/config.py:14) `public_base_url` 在部署时**必须**改

> **不改源码**，部署时通过环境变量覆盖：
>
> ```bash
> export MMR_PUBLIC_BASE_URL="http://<LAN_HOST>:8000"
> export MMR_ALGORITHM_GATEWAY_URL="http://127.0.0.1:18080"   # 算法与后端同机
> ```

#### ④ 启动命令

把 `uvicorn` 监听绑定到全网卡（沿用配置项）：

```bash
uvicorn app.main:app --host 0.0.0.0 --port 8000
```

### 3.2 算法端：保持 `0.0.0.0` 监听，明确 checkpoint

`algorithm/launcher/run_all.py` 当前已经是 `0.0.0.0`，**不需要改代码**。仅需：

- 在部署机上确认 `I2I_CKPT_PATH` / `T2I_CKPT_PATH` / `T2V_CKPT_PATH` 三个常量指向真实可用的权重路径（[`algorithm/launcher/run_all.py:11-13`](algorithm/launcher/run_all.py:11)）。
- 因为 gateway 与 3 个 worker **必须同机**（gateway 写死 `127.0.0.1:18081/2/3`，见 [`algorithm/gateway/app.py:11`](algorithm/gateway/app.py:11)），所以**算法 4 个进程整体必须部署在同一台局域网机**。

> 可选优化（非必需）：把 [`algorithm/gateway/app.py:11-15`](algorithm/gateway/app.py:11) 的 `WORKERS` 字典改为读环境变量 `ALG_I2I_URL` 等，便于将来拆机。本期不改。

### 3.3 前端：让 axios 能直连局域网后端

#### ① [`frontend/.env.example`](frontend/.env.example:1) 新增条目

```env
# 用于 Vite dev server 反向代理（npm run dev）
VITE_BACKEND_PROXY_TARGET=http://<LAN_HOST>:8000

# 用于打包后或直连模式下 axios 直接发到的后端地址
# 留空则走相对路径 '/'（依赖 Vite 代理或同源部署）
VITE_API_BASE_URL=http://<LAN_HOST>:8000
```

> 演示机本地运行 `npm run dev` 时，把这两项写入 `frontend/.env.local`（gitignored）即可。

#### ② [`frontend/vite.config.ts`](frontend/vite.config.ts:14) 让 dev server 监听全网卡（可选）

```ts
server: {
  host: '0.0.0.0',
  port: 5173,
  proxy: {
    '/api': { target: backendTarget, changeOrigin: true },
  },
},
```

> 开 `host: '0.0.0.0'` 后，同事可以用浏览器访问 `http://<演示机IP>:5173`，便于多人围观。

#### ③ axios 客户端无需改源码

[`frontend/src/api/client.ts:3`](frontend/src/api/client.ts:3) 已经是 `import.meta.env.VITE_API_BASE_URL || '/'`，符合预期。

### 3.4 关键约束：媒体文件可访问性

> 这是部署 LAN 时**最容易踩坑**、必须严格执行的一条。

后端在 `prepare_service` 里把媒体文件复制到本机 `data/assets/{scene}/{store_id}/...` 后，**把这个绝对路径作为 `key` 调算法 `/encode`**。算法 worker 读不到该路径就会失败。

因此本期**强约束**：

- ✅ **算法 4 个进程与后端必须部署在同一台 LAN 服务器、同一个文件系统**。
- ✅ 该机器的 `MMR_MANAGED_ASSETS_DIR`（默认 `backend/data/assets`）对算法进程可读。
- ✅ 推荐将 `backend/data/` 整体放在 SSD，并保证 ffmpeg/ffprobe 可执行（视频编码探测）。

如果未来需要拆成两台机器，需要后续追加方案：

1. 把 `data/assets` 通过 NFS / SMB 共享给算法机，并在算法机用同一路径挂载；或
2. 把 `key` 从“绝对路径”改成 `object_key`，让算法侧通过 HTTP 拉 `/api/v1/media/preview?object_key=...`（需要算法侧改造，本期不做）。

### 3.5 部署脚本：替代 `run.sh`

新增 [`run_lan.sh`](run_lan.sh)（或保留两份：`run.sh` 本机调试 + `run_lan.sh` 局域网）：

```bash
#!/usr/bin/env bash
set -e

# ------- 在 LAN 服务器上执行 -------
export MMR_PUBLIC_BASE_URL="http://$(hostname -I | awk '{print $1}'):8000"
export MMR_ALGORITHM_MODE="http"
export MMR_ALGORITHM_GATEWAY_URL="http://127.0.0.1:18080"
export MMR_CORS_ALLOW_ORIGINS='["*"]'

# 1) 启动算法网关 + 三个 worker（独立进程组，便于后续 kill）
( cd algorithm && nohup python launcher/run_all.py > ../logs/algorithm.log 2>&1 & echo $! > ../logs/algorithm.pid )

# 2) 启动后端
( cd backend && nohup uvicorn app.main:app --host 0.0.0.0 --port 8000 \
    > ../logs/backend.log 2>&1 & echo $! > ../logs/backend.pid )

echo "LAN services started. Backend at http://0.0.0.0:8000  Algorithm at http://0.0.0.0:18080"
```

演示机侧：

```bash
cd frontend
cp .env.example .env.local
# 编辑 .env.local，把 LAN_HOST 改为局域网服务器 IP
npm install
npm run dev -- --host 0.0.0.0
```

## 5. 风险与回滚

| 风险 | 触发条件 | 缓解 |
| --- | --- | --- |
| `preview_url` 仍然是 `127.0.0.1` | 忘记导出 `MMR_PUBLIC_BASE_URL` | `run_lan.sh` 自动注入；`/api/v1/health` 之外加一条断言巡检 |
| 跨域被浏览器拦截 | `MMR_CORS_ALLOW_ORIGINS` 未配置 | 默认 `["*"]`；正式发布前再收敛 |
| 算法读不到 `data/assets/**` | 后端、算法不同机 | 本期约定同机部署；如必须分机 → 走 NFS 挂载或后续改 `object_key` 方案 |
| 视频预览黑屏 | 编码非 H.264/MP4 / Range 处理 | 与原文档一致，不在本次范围；可后续加封面图 |

回滚：所有改动均通过**环境变量**和**少量 `main.py` 中间件配置**完成，恢复成本 = 取消环境变量 + 注释 `add_middleware`。

## 7. 后续可扩展项（不在本期范围）

1. 把算法 gateway → worker 的硬编码 `127.0.0.1` 改为环境变量，支持 worker 跨机。
2. 把 `prepare_service` 中传给算法的 `key` 从绝对路径改为 `object_key`，并由算法侧通过 `/api/v1/media/preview` 拉文件，彻底解耦文件系统。
3. 将 `local_thread_pool` 的任务队列替换为 Celery + Redis，配合多机部署。
4. 接入 Nginx 同源反代，把前端静态资源也放到 LAN 服务器，演示机不再起 dev server。
